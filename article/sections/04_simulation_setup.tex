\section{Simulation Setup}

The Rabot outdoor scenario is defined in \texttt{scenarios/rabot.yaml}. The scene is derived from an \gls{osm} boundary around the KU Leuven Gent Campus Rabot environment. Candidate distributed-\gls{ap} positions are generated along building walls at low height, one rooftop proxy is reserved for the central reference strategy, and user mobility is simulated over the outdoor walkable area with mixed pedestrian and bicycle profiles.

The simulation workflow used by the paper matches the repository workflow:
\begin{enumerate}
    \item build the outdoor scene and walk graph from \gls{osm},
    \item generate wall-mounted candidate \glspl{ap} and the single rooftop proxy site,
    \item simulate user trajectories over the walkable outdoor area,
    \item compute path-based \gls{csi} and explicit \gls{sinr} terms,
    \item compare the central, fixed-distributed, and three movable-distributed strategies,
    \item export reusable result files,
    \item generate analysis figures in a separate postprocessing step.
\end{enumerate}

This section documents the reproducible processing chain from scene generation to postprocessed figure export.

The figure set in \cref{sec:results_geometry,sec:results_sinr,sec:results_relocation} is bundled locally under \texttt{article/figures/rabot/} so the manuscript carries its own buildable asset set. These files are mirrored from the generated Rabot outputs in \texttt{outputs/rabot/} and \texttt{outputs/rabot/postprocessing/}, which remain the regeneration source for the reported figures.

For the current Rabot instantiation, the relocation controller starts from a single seed constellation drawn from the wall-mounted candidate pool using the configured random seed. That seed constellation is frozen for the \texttt{distributed\_fixed} baseline and also provides the initial labeled movable-\gls{ap} positions for \texttt{distributed\_movable}, \texttt{distributed\_movable\_optimization\_2}, and \texttt{distributed\_movable\_optimization\_3}. Thereafter, every relocation window applies the greedy subset search from \cref{sec:system_model} with the selected optimization objective and then solves a minimum-displacement assignment so that schedule traces, relocation distances, and move fractions correspond to the motion of persistent AP identities rather than to arbitrary reordering of candidate sites. For optimization~2 and optimization~3, the dormant candidate-position \gls{ap}-to-\gls{ue} channel is never queried during this search; only the measured \gls{ue}-to-\gls{ue} proxy channel or its power is used, and the actual deployed \gls{ap}-to-\gls{ue} \gls{csi} is recomputed only after the new candidate subset has been selected.
